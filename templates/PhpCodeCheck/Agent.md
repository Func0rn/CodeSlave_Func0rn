# Agent.md

## 目标

让 Codex 在 PHP 白盒代码审计任务中持续工作、持续下钻、持续产出，尽可能不因小问题中断，也不要频繁把问题抛回给人工。

## 工作原则

1. 默认直接开始，不等待人工继续确认。
2. 默认先自行收集上下文，再做判断，不要先提问。
3. 遇到信息不完整时，先基于代码仓库现状做合理假设并继续推进。
4. 只有在以下情况才允许打断人工：
   - 缺失源码、目录、依赖或关键输入，导致无法继续
   - 发现高风险破坏性操作，需要人工授权
   - 多种方案差异极大，且无法从代码中判断哪种是目标
5. 除上述阻塞场景外，不要把流程停在“分析完毕，等待你确认”。
6. 每次工作都要尽量推进到“已产出可用结果”而不是“给出建议”。

## 持续工作要求

Codex 执行任务时必须遵循以下要求：

- 连续推进任务，直到完成审计、生成报告、列出风险、给出修复建议。
- 如果某一步无法完全确认，先输出 `待验证`，再继续后续步骤，而不是中断。
- 如果 `trace` 不闭合、证据不足、分支不可达，应保留风险条目并降级，不得静默忽略。
- 优先从代码、配置、路由、控制器、服务层、模板层、依赖清单中自行提取信息。
- 优先复用已安装的 PHP 审计 skills，不要每次重新发明流程。
- 每完成一个阶段，就立刻进入下一个阶段，不要等待人工下达新指令。

## 对 Codex 的行为约束

1. 不要频繁提问。
2. 不要把简单决策抛给人工。
3. 不要只给分析思路而不落地执行。
4. 不要因为单个漏洞类型证据不足就停止整个审计。
5. 不要只扫关键字；要继续做 Source -> Sink 数据流追踪。
6. 不要只报结论；要给位置、证据链、影响、PoC、修复建议。
7. 不要漏掉“未确认但高风险”的问题；应归入 `待补证风险池`。

## 默认执行流程

当用户要求审计 PHP 项目时，Codex 默认按以下顺序持续执行：

1. 识别项目技术栈、目录结构、入口文件、框架类型
2. 调用 `php-audit-pipeline` 作为主入口
3. 进行路由枚举与参数建模：`php-route-mapper`
4. 进行鉴权映射：`php-auth-audit`
5. 扫描 Composer 依赖漏洞：`php-vuln-scanner`
6. 如果检测到框架，则调用对应专项 skill
7. 进行路由级数据流追踪：`php-route-tracer`
8. 按漏洞类型调用专项审计 skill
9. 汇总利用链：`php-exploit-chain-audit`
10. 生成完整 Markdown 审计报告
